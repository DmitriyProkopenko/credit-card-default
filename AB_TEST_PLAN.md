# План A/B-тестирования модели скоринга

## 1. Концепция теста
- **Цель:** Оценить, улучшает ли новая модель (v2, например, GradientBoosting) качество прогнозирования по сравнению с текущей базовой моделью (v1, RandomForest).
- **Гипотеза:** Модель v2 увеличит F1-score на 5% и снизит количество ложноположительных срабатываний (False Positives), что позволит одобрить больше надежных клиентов.
- **Разделение трафика:** Случайное распределение на уровне запросов (или пользователей) с использованием заголовка или cookie: 50% трафика на v1 (Control), 50% на v2 (Treatment).
- **Продолжительность:** 2 недели (для сбора статистически значимой выборки и учета недельной сезонности).

## 2. Метрики оценки
- **Основная техническая метрика:** F1-score для класса дефолта (1). Балансирует Precision и Recall.
- **Дополнительная метрика:** **Recall (Полнота)**. *Обоснование:* В кредитном скоринге пропуск дефолта (False Negative) обходится банку дороже, чем ложный отказ надежному клиенту (False Positive). Поэтому мы хотим максимизировать долю реально выявленных дефолтных клиентов.
- **Бизнес-метрика:** Ожидаемые финансовые потери (Expected Loss).

## 3. Статистический анализ
- **Статистический тест:** Двухвыборочный Z-test для пропорций (так как мы сравниваем доли успешных предсказаний / дефолтов в двух больших независимых группах).
- **Доверительные интервалы:** Строятся 95% доверительные интервалы для разницы метрик (v2 - v1).
- **Критерий успешности:** 
  1. p-value < 0.05 (отвергаем нулевую гипотезу об отсутствии разницы).
  2. Нижняя граница 95% доверительного интервала для прироста F1-score > 0 (или > минимального практического эффекта, MDE).
  3. Метрика Recall не ухудшилась статистически значимо.

## 4. Практическая реализация в сервисе
Для демонстрации в API добавлено поле `"version": "v1"`. В реальной системе балансировщик нагрузки (например, NGINX или Istio) маршрутизирует запросы на разные pod-ы (`ml-service-v1` и `ml-service-v2`) на основе веса (weight 50/50), а система сбора метрик (Prometheus) агрегирует фидбек (реальный дефолт через месяц) для расчета метрик по группам.
