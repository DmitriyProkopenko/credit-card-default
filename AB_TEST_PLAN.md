# План A/B-тестирования моделей скоринга

## 1. Концепция A/B-теста

### Цель
Оценить, улучшает ли новая модель (v2: Logistic Regression) качество прогнозирования по сравнению с текущей базовой моделью (v1: Random Forest).

### Гипотеза
**H1:** Модель v2 увеличит F1-score на 3% и снизит количество ложноположительных срабатываний (False Positives), что позволит одобрить больше надежных клиентов без увеличения риска.

**H0 (нулевая):** Разницы в метриках между v1 и v2 нет.

### Дизайн теста

**Разделение трафика:**
- **Метод:** Случайное распределение на уровне запросов (random assignment)
- **Пропорция:** 50% трафика на v1 (Control), 50% на v2 (Treatment)
- **Реализация:** В коде `src/app/api.py` используется `random.choice([model_v1, model_v2])`
- **Unit of randomization:** Отдельный запрос (request-level randomization)

**Продолжительность теста:**
- **Минимум:** 2 недели (14 дней)
- **Обоснование:** 
  - Учет недельной сезонности (рабочие дни vs выходные)
  - Сбор статистически значимой выборки (минимум 1000 запросов на группу)
  - Время для проявления эффекта (дефолт происходит в следующем месяце)
**Критерии остановки:**
- **Ранняя остановка:** Если p-value < 0.01 и эффект > 5% (clear winner)
- **Плановая остановка:** Через 14 дней или при достижении 10,000 запросов на группу

## 2. Метрики оценки

### 2.1. Основная метрика: F1-score для класса дефолта

**Определение:** Гармоническое среднее Precision и Recall для класса 1 (дефолт).

**Формула:**
```
F1 = 2 * (Precision * Recall) / (Precision + Recall)

где:
- Precision = TP / (TP + FP)  # Точность предсказания дефолта
- Recall = TP / (TP + FN)     # Полнота выявления дефолта
```
**Обоснование выбора:**
- Балансирует Precision и Recall
- Фокус на классе дефолта (миноритарный класс)
- Учитывает как ложные срабатывания, так и пропуски

**Целевое значение:** F1 >= 0.65 (текущий baseline: ~0.60)

### 2.2. Дополнительная метрика: Recall (Полнота)

**Определение:** Доля реально дефолтных клиентов, которых модель правильно идентифицировала.

**Формула:**
```
Recall = TP / (TP + FN)
```
**Обоснование выбора:**
В кредитном скоринге **пропуск дефолта (False Negative) обходится банку дороже**, чем ложный отказ надежному клиенту (False Positive):
- **False Negative:** Банк выдал кредит дефолтному клиенту → потеря всей суммы кредита
- **False Positive:** Банк отказал надежному клиенту → потеря комиссии, но не капитала

Поэтому мы хотим **максимизировать Recall** при условии, что Precision не падает ниже критического порога (85%).

**Целевое значение:** Recall >= 0.70 (текущий baseline: ~0.65)

### 2.3. Бизнес-метрики

**1. Expected Loss Reduction (Снижение ожидаемых потерь):**
```
EL = Σ(credit_amount_i * default_probability_i) для одобренных заявок
```
Цель: Снижение EL на 10% по сравнению с v1.

**2. Approval Rate при Precision >= 85%:**
```
Approval_Rate = approved / total * 100%
```
Цель: Увеличение Approval Rate на 5% без снижения Precision.

## 3. Статистический анализ

### 3.1. Статистический тест

**Выбранный тест:** Двухвыборочный Z-test для пропорций (Two-sample Z-test for proportions)

**Обоснование:**
- Большие выборки (n > 1000 на группу) → нормальное распределение метрик
- Сравнение долей (Precision, Recall, F1) между двумя независимыми группами
- Z-test более мощный, чем t-test, для больших выборок

**Формула Z-statistic:**
```
Z = (p2 - p1) / sqrt(p_pool * (1 - p_pool) * (1/n1 + 1/n2))
где:
p1, p2 — метрики (F1, Recall) для v1 и v2
n1, n2 — размеры выборок
p_pool = (x1 + x2) / (n1 + n2) — объединённая пропорция
```

### 3.2. Доверительные интервалы

**Метод:** 95% доверительный интервал для разницы метрик (v2 - v1)

**Формула:**
```
CI = (p2 - p1) ± Z_α/2 * SE

где:
- Z_α/2 = 1.96 (для 95% CI)
- SE = sqrt(p1*(1-p1)/n1 + p2*(1-p2)/n2)
```

**Интерпретация:**
- Если CI не содержит 0 → статистически значимая разница
- Если нижняя граница CI > 0 → v2 значительно лучше v1
- Если CI содержит 0 → разница не значима

### 3.3. Критерий успешности теста

**Условия для признания v2 успешной:**

1. **Статистическая значимость:**
   - p-value < 0.05 (уровень значимости α = 0.05)
   - 95% CI для разницы F1 не содержит 0

2. **Практическая значимость (Minimum Detectable Effect):**
   - Прирост F1 >= 3% (MDE = 0.03)
   - Прирост Recall >= 2% (MDE = 0.02)

3. **Бизнес-ограничения:**
   - Precision не ухудшился статистически значимо (p-value > 0.05 для разницы Precision)
   - Approval Rate не снизился более чем на 2%

**Решение:**
- **Внедрить v2:** Если все условия выполнены
- **Продлить тест:** Если p-value между 0.05 и 0.10 (trend)
- **Отклонить v2:** Если p-value > 0.10 или Precision ухудшился

## 5. Риски и ограничения
### 5.1. Риски
1. `Simpson's Paradox`: Разные распределения признаков в группах v1 и v2
  - *Митигация*: Проверка баланса ковариат (A/A тест перед A/B)
2. `Network Effect`: Клиенты могут взаимодействовать (например, семейные заявки)
  - *Митигация*: Randomization на уровне пользователя, а не запроса
2. `Novelty Effect`: Временное улучшение из-за новизны модели
  - *Митигация*: Минимальная длительность теста 2 недели

## 5.2. Ограничения
1. `Delayed Feedback`: Дефолт происходит через месяц → метрики рассчитываются с задержкой.
2. `Sample Size`: Требуется минимум 1000 запросов на группу для статистической мощности
3. `External Factors`: Изменение экономической ситуации может повлиять на результаты

## 6. Checklist для запуска A/B-теста
- Модели v1 и v2 обучены и сохранены
- A/A тест проведён (проверка баланса групп)
- Минимальный размер выборки рассчитан (power analysis)
- Логирование настроено (JSON формат)
- Метрики определены и согласованы с бизнесом
- Критерии остановки утверждены
- План действий при ранней остановке согласован
